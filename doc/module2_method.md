## 方法：多样性感知的 Actor Configuration 选择模块

### 方法描述

在 EvoTest 中，测试时学习过程由 Actor Agent 与 Evolver Agent 交替完成。Actor Agent 在当前 episode 中依据给定的 agentic configuration 与环境交互；episode 结束后，Evolver Agent 分析完整轨迹，并生成若干候选 child configurations，用于改进下一轮 Actor 的行为。原始 EvoTest 将 configuration 表示为策略提示、部署时记忆、推理超参数和工具调用例程的组合，并在父配置与候选子配置之间通过 UCB 进行选择。该机制能够在历史表现与探索之间取得一定平衡，但它仍然存在一个关键问题：Evolver 生成的多个候选配置往往高度同质化，UCB 虽然在形式上选择了不同配置，但这些配置在实际行为上可能只是同一策略的轻微改写。因此，原始选择器可能无法充分利用 Evolver 生成的候选空间，也容易在局部有效但全局不足的策略附近停滞。

为解决这一问题，我们在 EvoTest 的 Evolver-to-Actor configuration selection 阶段引入一个多样性感知的配置选择模块，记为 Diversity-Aware Configuration Selector。该模块不改变 Actor 的决策接口，也不改变 Evolver 的候选生成方式，而是插入在“候选配置生成”与“下一轮 Actor 配置确定”之间，用于重新评估、筛选和路由候选 actor configurations。其核心思想是：一个好的下一轮配置不仅应该在历史上表现较好，还应该与当前 episode 暴露出的失败模式相关，并且与其他候选配置保持足够的策略差异。

具体而言，在第 e 个 episode 结束后，Evolver 会基于轨迹生成一组候选配置。我们将父配置、Evolver 新生成的子配置，以及历史上曾取得较好表现的归档配置共同组成候选配置池。对于池中的每个配置，系统构建一个 configuration profile。该 profile 不只包含 prompt 的语义表示，还包含该配置携带的成功记忆、失败记忆、工具调用规则、超参数设置，以及该配置在过去 episode 中适用或失效的状态模式。因此，配置相似性不再被简单地定义为 prompt 相似性，而是被定义为完整 agentic configuration 在策略、记忆、工具使用和行为倾向上的综合相似性。

为了评估每个候选配置与当前任务的相关性，系统从历史轨迹中构造一组 proxy states。proxy states 是对过往经验的压缩表示，用于近似当前任务中最值得关注的关键决策场景。这些状态包括带来分数增长的成功状态、导致停滞的无效循环状态、重复执行失败动作的状态、出现 inventory 或位置变化的状态，以及靠近已知成功路径的中间状态。每个候选配置都会根据其 configuration profile 与这些 proxy states 的匹配程度被赋予相关性分数。相关性高的配置通常具有以下特征：能够继承过去有效的状态—动作经验，能够覆盖当前 episode 暴露出的失败模式，能够避免已知的重复无效动作，并且在当前游戏进展阶段具有合适的探索或利用倾向。

在相关性评估之后，模块进一步引入多样性感知筛选。受 relevance-weighted DPP 的启发，我们将候选配置选择建模为同时优化“任务相关性”和“配置多样性”的子集选择问题。相关性项鼓励选择与当前状态、当前失败模式和历史成功经验高度匹配的配置；多样性项则惩罚彼此过于相似的配置，避免候选集被大量等价 prompt、重复 memory 或相近 tool-use routine 占据。直观上，该机制倾向于保留一组互补配置：例如，一个配置偏向探索未知区域，一个配置偏向复现高分路径，一个配置专门避免导航循环，一个配置强化成功记忆调用。这样，后续选择器面对的不再是一组表面不同但行为接近的配置，而是一组覆盖不同策略方向的候选配置。

在完成多样性感知筛选后，系统并不直接丢弃原始 UCB 机制，而是将 UCB 从“全量候选池上的直接选择器”改造为“多样化候选池上的最终路由器”。也就是说，Diversity-Aware Configuration Selector 首先过滤掉明显冗余、低相关或高风险的候选配置，然后 UCB 在剩余的高质量候选集中进行最终选择。这样既保留了 UCB 对历史表现和不确定性的建模能力，又避免其探索项浪费在实际行为差异很小的配置之间。

进一步地，为了避免单纯依赖 episode-level 分数带来的噪声，模块在配置选择时引入上下文感知的评估信号。文本交互任务通常具有稀疏奖励特征，许多关键动作未必立即带来分数增长，例如检查物品、进入新房间、获得中间线索或改变 inventory 状态。因此，我们不只根据最终分数评估配置，还结合局部进展信号，例如是否发现新地点、是否改变 inventory、是否减少重复动作、是否触发新的有效 observation、是否接近历史成功前缀，以及是否避免已知失败状态。最终的配置价值估计由 episode-level return、proxy-state 相关性、局部进展信号和失败风险共同决定。这样，系统能够更早识别那些虽然尚未带来最终高分、但已经朝正确方向推进的候选配置。

该模块还包含一个轻量的 proxy-aligned configuration calibration 过程。Evolver 生成新的候选配置后，系统会在 proxy states 上检查该配置的行为倾向，而不是立即让其进入完整 episode。这个过程不更新模型参数，也不需要额外交互环境，只是利用已有轨迹中的代表性状态来判断候选配置是否存在明显风险。例如，若某个配置在已知失败状态中仍倾向于重复相同无效动作，或在已知成功状态中未能调用成功记忆，则该配置会被降权；若某个配置能够在多个 proxy states 上复现有效动作、规避失败循环并保持适度探索，则其相关性分数会提升。通过这种低成本校准，系统能够在正式执行前过滤一部分危险配置，从而降低 bad mutation 对后续 episode 的影响。

综上，加入该模块后的 EvoTest 在 configuration selection 阶段形成了一个新的选择流程：Evolver 仍然负责生成候选 actor configurations，但系统不再直接依赖 UCB 从原始候选池中选择下一轮配置，而是先构建 configuration profile，利用 proxy states 估计配置相关性和失败风险，再通过多样性感知机制获得更具策略覆盖度的候选集合，最后由 UCB 或上下文感知路由器确定下一轮 Actor 使用的 configuration。该模块只作用于 Evolver 选 Actor configuration 的位置，因此保持了 EvoTest 梯度无关、测试时自改进和整体配置演化的基本范式，同时增强了配置选择的稳定性和有效探索能力。

### 创新点

第一，本文将 EvoTest 中的 configuration selection 从单纯的 bandit-style 选择扩展为 relevance-aware and diversity-aware 的候选配置路由问题。原始 UCB 主要依据历史平均回报和探索奖励选择配置，而本文进一步显式建模候选配置与当前失败模式、当前进展阶段和历史成功经验之间的相关性，从而使配置选择更加贴合当前 episode 暴露出的具体问题。

第二，本文提出 configuration-level diversity modeling，将多样性建模从数据样本层面迁移到 agentic configuration 层面。不同于仅比较 prompt 文本差异，本文将 prompt、memory、hyperparameters、tool-use routines 以及 proxy-state 行为倾向共同纳入配置画像，使系统能够识别“表面不同但行为等价”的候选配置，并优先保留真正具有策略互补性的配置。

第三，本文引入 proxy states 作为配置选择的低成本校准信号。通过从历史轨迹中抽取成功状态、失败状态、循环状态和进展状态，系统可以在不进行额外环境交互的情况下评估新配置是否会复现成功经验或重复失败模式。这使得配置选择不再完全依赖稀疏的 episode-level return，而能够利用更细粒度的历史经验。

第四，本文将 UCB 从原始候选池上的直接选择器改造为多样化候选池上的最终选择器。该设计保留了 UCB 的探索—利用平衡优势，同时缓解了候选同质化导致的无效探索问题。换言之，UCB 的探索项不再浪费在多个近似等价的配置之间，而是在经过多样性感知筛选后的互补配置之间发挥作用。

第五，本文的模块是一个轻量的即插即用扩展。它不要求训练 backbone LLM，不改变 Actor 的交互接口，也不改变 Evolver 的候选生成格式，只在 Evolver 生成配置之后、Actor 执行下一轮 episode 之前进行选择增强。因此，该方法可以自然嵌入 EvoTest 的原始 Act-Evolve loop。

### 为什么该方法能够工作

该方法能够工作的核心原因在于，长程交互任务中的测试时学习瓶颈不仅来自“能否生成改进配置”，还来自“能否从多个候选配置中选出真正有用且互补的配置”。EvoTest 的 Evolver 已经具备根据轨迹生成新配置的能力，但生成多个候选并不保证这些候选具有足够差异。若候选配置高度相似，UCB 的探索机制实际上只能在同一个局部策略邻域中反复试探，难以发现新的有效行为模式。多样性感知筛选通过显式惩罚配置冗余，使候选池覆盖更多策略方向，从而提高有效探索概率。

其次，相关性建模使配置选择更加面向当前问题。一个配置的历史平均表现较高，并不意味着它适合当前状态或当前游戏阶段。例如，探索型配置可能在早期帮助发现地图，但在后期可能造成偏离；保守型配置可能在已知成功路径附近有效，但在未知区域可能过早收敛。通过引入 proxy states 和当前上下文相关性，系统能够判断一个配置是否真正适合当前 episode 暴露出的失败模式，而不是只依据过去的全局平均分。

第三，proxy-state 校准能够缓解稀疏奖励带来的选择噪声。在文本冒险游戏中，许多关键动作不会立即产生 reward，因此只用最终分数评价配置会丢失大量中间进展信息。proxy states 将历史轨迹中的关键状态提取出来，使系统能够检查候选配置是否在这些状态上做出合理行为。即使某个配置尚未在完整 episode 中取得高分，只要它在多个关键 proxy states 上表现出正确倾向，也可以被保留为潜在有用配置。

第四，该方法能够降低 bad mutation 的破坏性。Evolver 生成的 child configuration 可能包含有用改进，也可能引入错误规则、过度探索或不合理的工具使用策略。若直接将这些配置投入下一轮 episode，系统可能经历明显性能坠落。通过在配置执行前进行风险评估和多样性筛选，明显危险或冗余的 child configuration 会被降权，从而使学习曲线更加稳定。

最后，该方法与 EvoTest 的整体范式高度兼容。EvoTest 的优势在于不更新模型参数，而是在测试时演化整个 agentic system。本文提出的模块同样只操作 configuration 的选择与评估，不涉及梯度训练，因此不会破坏 EvoTest 的轻量性和通用性。它相当于为 Evolver 生成的候选配置增加了一个更强的选择器，使已有的配置演化过程能够被更充分、更稳健地利用。

### 局限性

第一，该模块依赖 proxy states 的代表性。如果早期 episode 的探索范围较窄，历史轨迹可能只覆盖少数失败模式，导致 proxy states 无法充分反映任务的真实状态空间。在这种情况下，配置选择器可能过度优化已观察到的局部问题，而无法发现尚未出现的高价值策略路径。

第二，配置相关性和多样性的估计仍然存在噪声。prompt、memory 和 tool-use routines 的文本差异不一定等价于实际行为差异；两个看似不同的配置可能在 Actor 执行时产生相似动作，两个语义接近的配置也可能因为 memory 或 hyperparameters 不同而表现出明显差异。因此，仅依赖文本表示是不够的，需要结合 proxy-state 上的行为响应来提高估计可靠性。

第三，该模块会带来额外计算开销。相比原始 UCB 选择，configuration profile 构建、proxy-state 评估、相似性计算和多样性筛选都需要额外的 embedding 计算或 LLM 判断。虽然这些成本通常低于完整 episode 交互，但在候选配置数量较多、proxy states 较多或 backbone 调用成本较高时，整体测试时学习开销会增加。

第四，多样性权重需要谨慎控制。早期 episode 中鼓励多样性有助于探索不同策略方向，但在后期已经发现稳定高分路径时，过强的多样性约束可能保留不必要的探索型配置，降低收敛速度。因此，该模块需要根据学习阶段动态平衡多样性与利用性：前期偏向探索和覆盖，后期偏向稳定复现高分路径。

第五，该方法不能完全解决奖励极端稀疏的问题。虽然 proxy states 和局部进展信号能够缓解只依赖最终分数的不足，但如果任务长期没有 reward、没有 inventory change、没有新地点发现，也缺少可识别的里程碑，那么配置评估仍然会面临信号不足的问题。在这种情况下，该模块需要与更强的状态抽象、进展检测或环境反馈解析机制结合使用。

第六，该模块只增强 Evolver 选择 Actor configuration 的环节，而不直接提升 Evolver 生成候选配置的质量。如果 Evolver 生成的所有候选配置都缺乏有效改进，选择器只能在有限候选中进行相对更优的筛选，无法凭空产生新的策略。因此，该模块的效果仍然受到 Evolver 候选生成能力的上限约束。

### 小结

本文提出的多样性感知 Actor Configuration 选择模块，专门作用于 EvoTest 中 Evolver 生成候选配置之后、Actor 执行下一轮 episode 之前的选择环节。该模块通过 configuration profile 建模完整配置，通过 proxy states 评估任务相关性和失败风险，通过多样性感知筛选减少候选同质化，并结合 UCB 或上下文路由完成最终配置选择。与原始 EvoTest 相比，该模块不改变测试时学习范式，却显著增强了候选配置选择的有效性，使系统能够在长程、稀疏奖励和高失败风险的交互任务中进行更稳健的自我改进。
